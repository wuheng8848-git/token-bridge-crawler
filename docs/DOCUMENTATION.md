# 文档规范

> 本规范适用于 Token Bridge Intelligence 项目，采用**渐进式披露**原则。

---

## 一、核心原则

### 1.1 渐进式披露

```
Level 0: 一句话定位     → README.md 第一行
Level 1: 一页纸概览     → index.md
Level 2: 快速参考       → docs/00-quick-reference.md
Level 3: 详细文档       → docs/01-07.md
Level 4: 深度资料       → 源码、迁移文件、决策记录
```

**读者路径**：
```
新人 → Level 0 → Level 1 → Level 2 → 按需深入
AI   → Level 1 → Level 2 → Level 3
```

### 1.2 文档分层

| 层级 | 文档 | 受众 | 更新频率 | 长度限制 |
|------|------|------|----------|----------|
| **入口层** | README.md, index.md | 所有人 | 低 | < 200 行 |
| **速查层** | 00-quick-reference.md | 开发者 | 中 | < 100 行 |
| **概念层** | 01-overview, 02-architecture | 新成员 | 中 | < 300 行 |
| **操作层** | 03-development, 04-deployment | 开发者 | 高 | < 400 行 |
| **参考层** | 05-api, 07-signal-model | 日常查阅 | 中 | 不限 |
| **记录层** | CHANGELOG.md, decisions/ | 架构师 | 低 | 按需增长 |

## 二、文件命名规范

### 2.1 docs/ 目录

```
docs/
├── 00-quick-reference.md      # 速查表（数字前缀决定排序）
├── 01-overview.md             # 项目概述
├── 02-architecture.md         # 架构设计
├── 03-development.md          # 开发指南
├── 04-deployment.md           # 部署文档
├── 05-api.md                  # API 文档
├── 06-roadmap.md              # 路线图
├── 07-customer-signal-model.md # 业务规则
└── decisions/                 # 架构决策记录（ADR）
    ├── 001-why-intelligence-system.md
    └── 002-translation-service-selection.md
```

**命名规则**：
- 数字前缀：`00-99`，决定文档顺序
- `00`：速查类
- `01-03`：概念类
- `04-05`：操作/参考类
- `06-07`：规划/业务类

### 2.2 根目录

```
根目录/
├── index.md           # 项目总览（AI/人类入口）
├── README.md          # 快速开始
└── CHANGELOG.md       # 版本更新记录
```

## 四、内容规范

### 4.1 标题层级

```
# 一级标题（文档标题，仅一个）
## 二级标题（主要章节）
### 三级标题（小节）
#### 四级标题（细节，少用）
```

**禁止**：五级及以上标题，应重构文档结构。

### 4.2 代码块

```markdown
# 命令示例
\`\`\`bash
go run ./cmd/intelligence -once
\`\`\`

# 代码示例
\`\`\`go
func main() {
    // ...
}
\`\`\`

# SQL 示例
\`\`\`sql
SELECT * FROM intelligence_items LIMIT 10;
\`\`\`
```

### 4.3 表格

```markdown
| 列1 | 列2 | 列3 |
|------|------|------|
| 值1 | 值2 | 值3 |
```

**规范**：
- 表头必须有
- 列数 ≤ 5，超过则拆分或用列表
- 长表格（> 20 行）考虑单独文件

### 4.4 链接

```markdown
# 内部链接（相对路径）
[index.md](./index.md)
[docs/05-api.md](./docs/05-api.md)

# 外部链接
[Keep a Changelog](https://keepachangelog.com/)
```

## 五、更新规范

### 5.1 必须更新

| 场景 | 更新文档 |
|------|----------|
| 新增功能 | CHANGELOG.md + 相关 docs/ |
| 架构变更 | 02-architecture.md + ADR |
| API 变更 | 05-api.md |
| 数据结构变更 | 05-api.md 数据存储章节 |
| 新增采集器/检测器 | 相关 docs/ |
| 版本发布 | CHANGELOG.md |

### 5.2 更新检查清单

```markdown
- [ ] 文档标题是否准确
- [ ] 最后更新日期是否修改
- [ ] 相关文档链接是否更新
- [ ] index.md 索引是否需要更新
- [ ] 代码示例是否可运行
- [ ] 数据统计是否最新
```

### 5.3 文档版本号

每个文档底部标注版本：

```markdown
**最后更新**：YYYY-MM-DD
**版本**：vX.Y
```

- **主版本（X）**：重大重构
- **次版本（Y）**：内容更新

## 六、AI 友好规范

### 6.1 入口提示

在 `index.md` 开头添加：

```markdown
> **AI 提示**：本文档为项目总览，建议先阅读本文档再深入其他文档。
```

### 6.2 结构化数据

关键数据用表格或列表，便于解析：

```markdown
## 当前状态

| 指标 | 数值 |
|------|------|
| 情报总数 | 770 |
| 翻译覆盖 | 100% |
```

### 6.3 明确的章节标题

避免模糊标题：

```
❌ 其他注意事项
✅ 常见错误处理

❌ 总结
✅ 数据存储现状
```

## 八、示例

### 8.1 好的文档结构

```
✅ index.md
├── 一、项目定位（3 句话）
├── 二、核心功能（表格）
├── 三、数据流向（ASCII 图）
├── 四、文档索引（表格）
└── 五、快速理解清单（列表）

✅ docs/05-api.md
├── 概述
├── 接口规范
├── 数据库表结构
├── 数据存储现状（统计 + 查询）
└── 更新日期
```

### 8.2 不好的文档结构

```
❌ README.md
├── 项目历史（无关内容）
├── 详细架构（应在 docs/）
├── 完整 API 文档（应在 docs/）
└── 500+ 行（太长）

❌ docs/05-api.md
├── 没有目录结构
├── 代码示例无法运行
├── 数据统计过时
└── 没有更新日期
```
